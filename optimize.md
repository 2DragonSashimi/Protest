# 구현 / 시뮬레이션 문제 최적화 패턴 정리

## 0. 최적화의 핵심 관점

구현 문제에서 최적화는 단순히 코드를 빠르게 고치는 것이 아니다.

대부분은 다음 과정을 거친다.

```text
브루트포스 풀이 작성
→ 반복되는 전체 탐색 발견
→ 실제로 영향을 받는 범위 확인
→ 상태를 다시 정의
→ 자료구조나 테크닉 적용
```

즉 최적화는 다음 질문에서 시작한다.

```text
이 계산을 매번 전체에 대해 다시 해야 하는가?
```

브루트포스는 문제를 이해하는 데 좋다.  
하지만 시간 초과가 나면, 브루트포스를 버리는 것이 아니라 **브루트포스의 병목을 분석해서 계산 범위를 줄이는 방향**으로 가야 한다.

---

# 1. 최적화 기본 절차

## 1.1 먼저 직관적인 풀이를 만든다

처음부터 최적화를 생각하면 코드가 꼬이기 쉽다.

먼저 가장 직접적인 상태와 로직으로 푼다.

```text
문제 조건 그대로 구현
전체 탐색
전체 재계산
직관적인 시뮬레이션
```

이 단계의 목적은 정답 구조를 이해하는 것이다.

---

## 1.2 병목을 찾는다

브루트포스를 작성한 뒤에는 반복 구조를 본다.

예를 들어:

```text
API 호출 수 × 후보 수 × 검사 비용
턴 수 × 전체 보드 검사 비용
이벤트 수 × 다익스트라 횟수
시간 범위 × 매초 갱신 비용
```

이런 식으로 쪼갠다.

질문:

```text
가장 자주 호출되는 함수는 무엇인가?
그 함수가 매번 전체를 훑는가?
가장 큰 곱셈 항은 무엇인가?
```

---

## 1.3 전체를 봐야 하는지 의심한다

병목이 보이면 바로 이 질문을 던진다.

```text
전체를 다시 볼 필요가 진짜 있는가?
```

대부분의 최적화는 이 질문에서 시작한다.

```text
전체 배열 → 바뀐 위치 주변만
전체 그래프 → 새로 추가된 노드/간선 기준만
전체 후보 → 영향을 받은 후보만
매 시간 갱신 → 조회 시점에 누적 계산
기존 그룹 유지 → 현재 상태 기준으로 재구성
```

# 2. 최적화 질문과 자료구조 매핑

## 2.1 존재 여부를 빠르게 확인해야 하는가?

예시:

```text
이미 방문했는가?
이미 사용한 source인가?
이미 요청한 파일인가?
이미 삭제된 객체인가?
```

사용 자료구조:

```python
set
```

예시:

```python
visited = set()
used_sources = set()
requested = set()
```

판단 기준:

```text
값이 있는지만 빠르게 확인하면 된다 → set
값에 대응되는 추가 정보가 필요하다 → dict
```

시간복잡도:

```text
x in set → 평균 O(1)
```

---

## 2.2 ID로 상태를 바로 찾아야 하는가?

예시:

```text
파일 ID로 크기 찾기
차량 번호로 주차 정보 찾기
빙하 ID로 방향/부피 찾기
컴퓨터와 파일 쌍으로 다운로드 상태 찾기
```

사용 자료구조:

```python
dict
```

예시:

```python
file_size[fid] = size
parking_info[car_no] = (time, zone, slot)
glacier_info[group_id] = {
    "move_dir": move_dir,
    "area": area,
    "volume": volume,
}
```

복합 key도 자주 사용한다.

```python
download_state[(computer, file_id)] = state
```

판단 기준:

```text
특정 ID를 받아서 즉시 상태를 찾아야 한다 → dict
```

---

## 2.3 최솟값 / 최댓값을 계속 꺼내야 하는가?

예시:

```text
가장 빨리 끝나는 이벤트
가장 작은 번호의 빈 슬롯
가장 비용이 작은 노드
가장 우선순위 높은 후보
```

사용 자료구조:

```python
heapq
```

예시:

```python
heapq.heappush(heap, item)
value = heapq.heappop(heap)
```

대표 사용:

```text
다익스트라
이벤트 시뮬레이션
최소 슬롯 관리
상위 후보 관리
```

주의:

```text
Python heapq는 min heap이다.
최댓값이 필요하면 음수로 넣는다.
```

예시:

```python
heapq.heappush(heap, (-score, id))
```

---

## 2.4 heap에서 중간 삭제가 필요한가?

heap은 중간 원소 삭제가 어렵다.

이럴 때는 보통 **lazy deletion**을 쓴다.

## Lazy Deletion

핵심:

```text
삭제할 때 heap에서 직접 지우지 않는다.
나중에 pop할 때 현재 상태와 비교해서 유효하지 않으면 버린다.
```

예시:

```python
heapq.heappush(heap, (time, id, token))
```

꺼낼 때:

```python
time, id, token = heapq.heappop(heap)

if current_token[id] != token:
    continue
```

사용 상황:

```text
이벤트 heap
검색용 heap
우선순위 큐에 오래된 값이 남는 경우
```

판단 기준:

```text
heap에 낡은 데이터가 남을 수 있다
직접 삭제가 어렵다
꺼낼 때 검증 가능하다
→ lazy deletion
```

# 3. 대표 최적화 패턴

# 3.1 Local Update

## 개념

전체를 다시 검사하지 않고, **바뀐 위치 주변만 검사**하는 방식이다.

```text
전체 검사
→ 변경된 위치 주변 검사
```

## 적용 조건

```text
1. 이전 상태가 안정적이다.
2. 이번 연산으로 바뀐 위치가 적다.
3. 새로 생기는 변화는 바뀐 위치를 포함해야 한다.
```

## 예시 사고

```text
두 칸을 교환했다.
값이 바뀐 곳은 두 칸뿐이다.
기존 상태에 문제가 없었다면,
새 문제는 그 두 칸 주변에서만 생긴다.
```

## 대표 형태

```python
changed = [(y1, x1), (y2, x2)]
check_around(changed)
```

## 자주 쓰는 테크닉

```text
changed cells 기록
affected rows/cols 기록
주변 4방향 / 8방향 검사
특정 좌표가 포함된 line만 검사
```

## 사용 예시 유형

```text
격자 swap
퍼즐
보드게임
한 칸 색칠
한 칸 삭제
주변 조건 검사
```

## 판단 질문

```text
이번 연산으로 바뀐 위치는 어디인가?
바뀌지 않은 곳에서 새 결과가 생길 수 있는가?
이전 상태가 정상 상태였는가?
```

# 3.2 Lazy Update

## 개념

시간이 흐를 때마다 매번 갱신하지 않고, **필요한 시점에 그동안의 변화를 한 번에 반영**하는 방식이다.

## 기본 공식

```python
value += rate * (now - last_time)
last_time = now
```

## 적용 조건

```text
1. 시간이 흐르며 값이 일정 속도로 변한다.
2. 중간 이벤트로 rate가 바뀔 수 있다.
3. 매 시간마다 갱신하면 너무 느리다.
4. 특정 이벤트나 조회 시점에만 값이 필요하다.
```

## 대표 상태

```python
state = [value, rate, last_time]
```

또는 source 개수에 따라 rate가 정해지는 경우:

```python
state = [downloaded, sources, last_time]
```

## 사용 예시 유형

```text
다운로드량
충전량
방전량
생산량
점수 누적
시간 기반 성장
쿨타임
대기 시간
```

## 판단 질문

```text
매 시간마다 실제로 갱신해야 하는가?
마지막 갱신 시각과 변화율만 알면 현재 값을 계산할 수 있는가?
rate가 바뀌는 시점만 기록하면 되는가?
```

# 3.3 Rebuild

## 개념

보조 상태를 계속 유지하기 어렵다면, **현재 실제 상태를 기준으로 다시 만드는 방식**이다.

```text
기존 라벨 유지
→ 복잡함

현재 상태 기준 재구성
→ 단순함
```

## 적용 조건

```text
1. 그룹이 쪼개질 수 있다.
2. 그룹이 합쳐질 수 있다.
3. 기존 ID를 유지하려고 하면 예외가 많다.
4. 입력 크기가 재구성 가능한 수준이다.
```

## 대표 예시

```text
grid에서 connected component 다시 찾기
BFS/DFS로 group_id 다시 붙이기
현재 상태 기준으로 group info 다시 계산
```

## 대표 형태

```python
components, group_id_grid = find_components(current_grid)
```

## 사용 예시 유형

```text
섬 나누기
빙하 분리/병합
블록 그룹 삭제
연결 컴포넌트 변화
격자 기반 군집 문제
```

## 판단 질문

```text
실제 상태는 무엇인가?
보조 상태는 무엇인가?
보조 상태를 계속 갱신하는 게 쉬운가?
아니면 실제 상태에서 다시 계산하는 게 쉬운가?
```

# 3.4 State Separation

## 개념

비슷해 보이는 상태라도 문제에서 역할이 다르면 분리해야 한다.

하나의 자료구조에 섞으면 조건이 꼬인다.

## 예시 상태 구분

```text
대기 중
진행 중
완료됨
삭제됨
기록만 남음
실제로 존재함
검색 대상임
이벤트 대상임
공유 가능함
소유만 함
```

## 대표 예시

```text
공유 파일
다운로드 중 파일
다운로드 완료 파일
```

이들은 모두 “파일을 가지고 있다”처럼 보일 수 있지만, 이후 이벤트에 참여하는 방식이 다르다.

따라서 분리한다.

```python
share_file[fid] = set()
downloading[com][fid] = ...
completed[com].add(fid)
```

## 판단 질문

```text
이 두 상태는 이후 이벤트에서 같은 역할을 하는가?
검색 대상도 같은가?
완료 후에도 같은 방식으로 처리되는가?
한쪽은 source가 되고 다른 쪽은 source가 안 되는가?
```

역할이 다르면 분리한다.

# 3.5 Viewpoint Shift

## 개념

반복의 기준을 바꿔서 계산량을 줄이는 방식이다.

```text
요청자 기준
→ source 기준

target 기준
→ 새로 추가된 객체 기준
```

## 적용 조건

```text
1. A마다 B를 모두 확인하고 있다.
2. 그런데 실제로 새로 변한 것은 B 하나 또는 일부다.
3. 새로 변한 객체 기준으로 보면 반복이 줄어든다.
```

## 예시 사고

```text
모든 요청자마다 새 source를 찾는다.
→ 새 source에서 한 번 탐색한 뒤 요청자들을 확인한다.
```

## 대표 형태

```python
dist = dijkstra(new_source)

for target in targets:
    if dist[target] <= LIMIT:
        update(target)
```

## 사용 예시 유형

```text
새 공급처 추가
새 감염원 추가
새 파일 추가
새 후보 추가
새 노드 추가
```

## 판단 질문

```text
지금 반복 기준이 맞는가?
새로 추가된 것 기준으로 보면 더 적게 볼 수 있는가?
source와 target을 뒤집을 수 있는가?
```

# 3.6 New Element Impact

## 개념

새로운 요소가 추가되었을 때, 새롭게 좋아지는 경우는 보통 그 새 요소를 포함한다.

```text
새 간선 추가
→ 새로 짧아진 경로는 그 간선을 지난다.
```

## 대표 예시: 새 edge 추가

새 간선이 `A --w-- B`로 추가되었다.

어떤 `u`, `v` 사이의 새 경로는 다음 중 하나이다.

```text
u → A → B → v
u → B → A → v
```

그래서 새 간선 추가 전 그래프에서:

```python
distA = dijkstra(A)
distB = dijkstra(B)
```

를 구하면, 새 edge를 사용하는 거리 후보를 빠르게 계산할 수 있다.

```python
distA[u] + w + distB[v]
distB[u] + w + distA[v]
```

## 적용 조건

```text
새로 추가된 요소가 명확하다.
새롭게 달라지는 경우가 그 요소를 반드시 포함한다.
그 요소를 기준으로 계산하면 전체 재계산을 피할 수 있다.
```

## 판단 질문

```text
새로 좋아지는 경우는 새 요소를 반드시 사용하는가?
새 요소의 양 끝점 / 위치 / ID 기준으로 계산할 수 있는가?
```

# 3.7 Caching / Memoization

## 개념

같은 계산을 반복하면 결과를 저장해서 재사용한다.

## 기본 형태

```python
cache[key] = result
```

상태가 바뀌는 문제에서는 version을 같이 둔다.

```python
cache[key] = (version, result)
```

상태 변경 시:

```python
version += 1
cache.clear()
```

또는 특정 key만 무효화한다.

## 적용 조건

```text
1. 같은 입력으로 같은 계산이 반복된다.
2. 결과가 일정 기간 유효하다.
3. 무효화 시점이 명확하다.
```

## 사용 예시 유형

```text
같은 시작점 다익스트라
같은 구간 계산
같은 상태 평가
같은 후보 점수
```

## 판단 질문

```text
같은 계산을 반복하고 있는가?
그 계산 결과는 언제까지 유효한가?
상태 변화 시 캐시를 어떻게 지울 수 있는가?
```

주의:

```text
캐시 무효화가 애매하면 코드가 더 복잡해질 수 있다.
```

# 3.8 Priority Structure

## 개념

매번 정렬하지 않고, 필요한 우선순위를 자료구조로 유지한다.

## 사용 자료구조

```text
최솟값 / 최댓값 → heap
존재 여부 → set
ID 기반 조회 → dict
정렬된 순서 유지 → balanced tree 계열
삭제 어려움 → lazy deletion
```

## tuple key

여러 조건의 우선순위는 tuple로 표현한다.

```python
key = (-score, y, x)
```

예시:

```python
best = min(
    candidates,
    key=lambda x: (
        -x.volume,
        x.area,
        x.y,
        x.x,
    )
)
```

의미:

```text
volume 큰 것 우선
area 작은 것 우선
y 작은 것 우선
x 작은 것 우선
```

## 판단 질문

```text
매번 정렬하고 있는가?
상위 몇 개만 필요한가?
최솟값/최댓값만 반복해서 필요한가?
우선순위 조건이 여러 개인가?
```

# 3.9 Limited Search

## 개념

탐색 범위에 제한이 있으면 끝까지 탐색하지 않는다.

## 예시: 거리 제한 다익스트라

```python
if cur_dist > LIMIT:
    break
```

```python
if next_dist > LIMIT:
    continue
```

## 적용 조건

```text
목표 범위가 제한되어 있다.
LIMIT 밖의 결과는 필요 없다.
```

## 사용 예시 유형

```text
거리 5000 이하
깊이 K 이하
시간 T 이하
비용 C 이하
```

## 판단 질문

```text
탐색 결과 전체가 필요한가?
특정 거리/깊이/비용 이하만 필요한가?
LIMIT을 넘으면 더 확장할 필요가 없는가?
```

# 3.10 Union-Find vs BFS Rebuild

## Union-Find가 좋은 경우

```text
합치기만 많다.
삭제나 분리가 없다.
연결 여부만 빠르게 묻는다.
```

대표 연산:

```python
union(a, b)
find(x)
```

## BFS Rebuild가 좋은 경우

```text
삭제가 있다.
그룹이 분리될 수 있다.
현재 모양 기준으로 다시 찾아야 한다.
grid 크기가 크지 않다.
```

판단 기준:

```text
합치기만 있음 → Union-Find
삭제/분리도 있음 → BFS/DFS rebuild
```

# 4. 구현 문제에서 자주 쓰는 자료구조 요약

| 목적 | 자료구조 / 테크닉 |
|---|---|
| 존재 여부 확인 | `set` |
| ID로 상태 조회 | `dict` |
| 복합 상태 조회 | tuple key dict |
| 최솟값/최댓값 | `heapq` |
| 오래된 heap 데이터 처리 | lazy deletion, token |
| 시간 누적 | lazy update, last_time |
| 격자 연결 그룹 | BFS/DFS |
| 그룹 합치기만 | Union-Find |
| 그룹 분리/병합 | BFS rebuild |
| 우선순위 여러 조건 | tuple key |
| 전체 후보 중 일부만 영향 | local update |
| 같은 계산 반복 | cache, version |
| 거리 제한 | limited BFS/Dijkstra |
| 새 edge 추가 영향 | endpoint distance |
| 상태 의미가 다름 | state separation |

# 5. 브루트포스에서 최적화로 가는 체크리스트

브루트포스 풀이 후 아래 질문을 순서대로 확인한다.

## 5.1 병목 확인

```text
가장 자주 호출되는 함수는 무엇인가?
그 함수는 전체 배열/그래프/객체를 훑는가?
시간복잡도에서 가장 큰 곱셈 항은 무엇인가?
```

---

## 5.2 변화 범위 확인

```text
이번 연산으로 실제로 바뀐 위치는 어디인가?
새 결과가 바뀐 위치 주변에서만 생기는가?
이전 상태가 안정 상태였는가?
```

가능하면:

```text
Local Update
```

---

## 5.3 시간 흐름 확인

```text
시간이 흐르면서 값이 일정하게 변하는가?
매 시간 갱신이 필요한가?
마지막 갱신 시각과 변화율로 현재 값을 계산할 수 있는가?
```

가능하면:

```text
Lazy Update
```

---

## 5.4 상태 유지 vs 재구성

```text
보조 상태를 계속 유지하는 것이 쉬운가?
그룹이 쪼개지거나 합쳐지는가?
현재 상태에서 다시 BFS/스캔하는 것이 더 단순한가?
```

가능하면:

```text
Rebuild
```

---

## 5.5 상태 의미 분리

```text
비슷해 보이는 상태들이 실제로 같은 역할을 하는가?
한쪽은 이벤트에 참여하고 다른 쪽은 참여하지 않는가?
검색 대상이 다른가?
```

필요하면:

```text
State Separation
```

---

## 5.6 반복 기준 전환

```text
지금 반복 기준이 맞는가?
source와 target을 바꿀 수 있는가?
새로 추가된 요소 기준으로 보면 더 적게 볼 수 있는가?
```

가능하면:

```text
Viewpoint Shift
```

---

## 5.7 새 요소 영향 확인

```text
새로 좋아지는 경우가 새 요소를 반드시 포함하는가?
새 edge / node / object 기준으로만 확인할 수 있는가?
```

가능하면:

```text
New Element Impact
```

---

## 5.8 중복 계산 확인

```text
같은 입력으로 같은 계산을 반복하고 있는가?
결과가 언제까지 유효한가?
무효화 시점이 명확한가?
```

가능하면:

```text
Caching
```

```text
3. 각 API/턴/이벤트가 바꾸는 것은 무엇인가?

6. 변화의 영향 범위는 어디까지인가?
7. 시간이 흐르는 값은 lazy하게 계산할 수 있는가?
8. 그룹 상태는 유지할 것인가, 재구성할 것인가?
9. 상태들이 의미별로 분리되어 있는가?
10. 반복 기준을 바꿀 수 있는가?
11. 새로 추가된 요소만 보면 되는가?
12. 캐싱이 가능한가?
13. 우선순위 자료구조가 필요한가?
```

# 7. 브루트포스와 최적화의 관계

브루트포스는 나쁜 풀이가 아니다.

브루트포스는 다음을 알려준다.

```text
필요한 상태
정확한 규칙
반복되는 작업
시간 초과가 나는 위치
```

최적화는 브루트포스를 버리는 것이 아니라,  
브루트포스에서 반복되는 전체 탐색을 줄이는 것이다.

```text
브루트포스:
    모든 것을 매번 계산한다.

최적화:
    바뀐 것만 계산한다.
    필요한 시점에만 계산한다.
    유지하기 어려운 보조 상태는 다시 만든다.
    의미가 다른 상태는 분리한다.
    반복 기준을 바꾼다.
```
