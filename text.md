주차장 문제 풀이 설명

1. 문제를 처음 봤을 때 해야 하는 해석

이 문제는 조건이 많아 보이지만, 실제로는 다음 3가지 기능으로 나눌 수 있다.

1) 차량의 입차와 출차
2) 시간 경과에 따른 자동 견인
3) 차량 번호 뒷 4자리 기준 검색

즉, 문제를 풀기 위해서는 단순히 차량을 저장하는 것만으로는 부족하고,
현재 주차 중인 차량 상태, 현재 견인된 차량 상태, 미래에 견인될 차량, 검색을 빠르게 하기 위한 인덱스를 각각 관리해야 한다.

--------------------------------------------------

2. 왜 차량 상태를 parked, towed 두 개의 dict로 나누는가

차량 번호 하나에 대해 현재 상태는 사실 3가지뿐이다.

1) 현재 주차 중
2) 현재 견인 기록 상태
3) 둘 다 아님

그래서 차량 번호를 key로 하여 바로 현재 상태를 확인할 수 있도록 dict를 두는 것이 가장 자연스럽다.

parked
현재 주차 중인 차량을 저장한다.

parked[car] = (ticket, enter_time, zone, slot)

여기서 필요한 정보는 다음과 같다.

- enter_time: 출차 시 주차 시간을 계산하기 위해 필요
- zone, slot: 출차나 견인 시 어떤 슬롯을 비워야 하는지 알기 위해 필요
- ticket: 같은 차량이 재입차할 수 있기 때문에 현재 주차 상태가 최신인지 구분하기 위해 필요

즉 parked는 단순히 “이 차량이 주차 중인가?”를 확인하는 용도가 아니라,
출차와 견인 처리에 필요한 실제 상태 정보를 저장하는 핵심 자료구조다.

towed
현재 견인 기록 상태인 차량을 저장한다.

towed[car] = (ticket, tow_time)

여기에는 tow_time만 있으면 충분하다.

견인된 차량의 출차 반환값은
-(주차된 기간 + 견인된 기간 * 5)
인데, 주차된 기간은 항상 최대 허용 시간 L로 고정이다.

즉 견인된 뒤에는 입차 시각을 다시 들고 있을 필요 없이
-(L + (mTime - tow_time) * 5)
로 바로 계산할 수 있다.

--------------------------------------------------

3. 왜 빈 슬롯 관리를 free_cnt와 free_slots로 나누는가

차량이 입차할 때 슬롯 선택 규칙은 두 단계로 이루어진다.

1) 빈 슬롯이 가장 많은 구역을 선택
2) 그 구역 안에서 번호가 가장 작은 빈 슬롯 선택

즉 이 문제는 “전체 슬롯 중 가장 작은 빈 슬롯”을 찾는 문제가 아니라,
구역 선택과 슬롯 선택이 분리된 문제다.

free_cnt[zone]
각 구역의 빈 슬롯 개수를 저장한다.

이 값은 “어느 구역이 가장 비어 있는가”를 판단할 때 사용된다.
구역 수 N은 최대 26이므로 매우 작다.
따라서 구역 선택은 매번 선형 탐색으로 해도 충분히 빠르다.

즉 여기서는 복잡한 우선순위 구조를 만들기보다,
단순한 배열이 가장 효율적이다.

free_slots[zone]
각 구역 안의 빈 슬롯 번호를 저장하는 min-heap이다.

이건 “선택된 구역에서 가장 번호가 작은 빈 슬롯”을 빠르게 찾기 위해 필요하다.

슬롯은 출차나 견인에 의해 중간 번호가 다시 비워질 수 있기 때문에,
단순히 증가하는 포인터로는 관리할 수 없다.

예를 들어 슬롯 000, 001, 002가 있었는데 001이 비워졌다면,
다음 입차는 반드시 001에 들어가야 한다.
이 상황은 min-heap으로 가장 자연스럽게 처리할 수 있다.

--------------------------------------------------

4. 왜 구역 전체는 heap으로 관리하지 않는가

처음 생각하면 “구역도 heap으로 관리하면 더 빠르지 않을까?”라고 생각할 수 있다.

하지만 구역 수는 최대 26개뿐이다.
이 정도 크기에서는 매번 선형 탐색하는 비용이 매우 작다.

오히려 구역까지 heap으로 관리하면

- 빈 슬롯 수가 바뀔 때마다 우선순위 갱신 필요
- heap 내부 임의 갱신이 어려워 lazy deletion 필요
- 구현 복잡도 증가

라는 문제가 생긴다.

즉 이 문제에서는

- 구역 선택: 단순 배열 + 선형 탐색
- 구역 내 슬롯 선택: heap

으로 역할을 분리하는 것이 가장 합리적이다.

--------------------------------------------------

5. 왜 자동 견인을 tow_heap으로 관리하는가

견인은 입차 직후 바로 일어나는 것이 아니라,
“입차 시각 + L”이 되는 미래 시각에 발생한다.

즉 견인은 미래 예약 이벤트로 볼 수 있다.

문제 조건에서 mTime은 호출될 때마다 증가하므로,
현재 시각까지 도달한 견인 이벤트만 순서대로 처리하면 된다.

이 구조는 전형적인 이벤트 시뮬레이션이고,
이럴 때 가장 적합한 자료구조가 min-heap이다.

tow_heap = [(tow_time, ticket, car)]

이렇게 두면 가장 먼저 견인될 차량부터 순서대로 꺼낼 수 있다.

--------------------------------------------------

6. 왜 ticket이 필요한가

처음에는 (tow_time, car)만 넣어도 될 것 같아 보인다.

하지만 차량은 재입차할 수 있다.

예를 들어,
100시에 입차
150시에 출차
200시에 다시 입차

했다면, 첫 번째 입차 시 생성한 견인 이벤트가 heap 안에 남아 있을 수 있다.

그 상태에서 나중에 그 이벤트가 나오면,
현재 주차 중인 두 번째 입차 상태와 혼동될 수 있다.

그래서 차량이 새로 주차될 때마다 증가하는 ticket을 부여하고,
parked[car]에 저장된 ticket과 heap에서 꺼낸 이벤트의 ticket이 일치할 때만 현재 유효한 이벤트로 인정한다.

즉 ticket은 같은 차량 번호가 여러 번 등장할 수 있는 상황에서
현재 상태와 과거 기록을 구분하기 위한 버전값이다.

--------------------------------------------------

7. 왜 suffix별 검색 인덱스가 필요한가

search(mTime, mStr)는 차량 번호 전체를 대상으로 하지 않고,
뒷 4자리 ZZZZ가 같은 차량만 대상으로 삼는다.

검색 횟수는 최대 50,000번이므로,
매번 parked 전체와 towed 전체를 훑어서 suffix를 비교하면 비효율적일 수 있다.

그래서 뒷 4자리별로 차량을 미리 묶어 두는 인덱스가 필요하다.

즉 검색을 빠르게 하기 위해
park_suffix[suffix]
tow_suffix[suffix]
라는 두 개의 구조를 둔다.

--------------------------------------------------

8. 왜 suffix별 자료구조도 heap인가

같은 suffix 안에서 차량의 우선순위는 다음과 같다.

1) 주차 중인 차량이 견인된 차량보다 우선
2) 같은 상태라면 XX가 작은 차량 우선
3) 그것도 같다면 Y가 빠른 차량 우선

이 우선순위를 보면,

- parked와 towed는 상태 우선순위가 다르므로 구조를 분리하는 것이 좋고
- 같은 상태 안에서는 차량 번호 기준 정렬만 되면 된다

따라서 suffix별로 parked용 heap, towed용 heap을 따로 두는 것이 가장 간단하다.

park_suffix[suffix] = heap of (car, ticket)
tow_suffix[suffix] = heap of (car, ticket)

--------------------------------------------------

9. 왜 같은 suffix 안에서는 차량 번호 문자열 자체로 비교해도 되는가

차량 번호 형식은 항상 XXYZZZZ이다.

search에서는 이미 ZZZZ가 같은 차량만 찾고 있으므로,
실제로 비교해야 하는 부분은 앞의 XXY이다.

문제의 우선순위는
- XX를 숫자로 봤을 때 작은 것
- 같으면 Y 알파벳이 빠른 것

인데, 이건 문자열 사전순과 정확히 일치한다.

예를 들어
67K7777
76Z7777
77G7777
77H7777

은 문자열 사전순으로 정렬해도 문제의 우선순위와 동일하다.

그래서 따로 복잡한 비교 함수를 만들 필요 없이,
heap 원소를 (car, ticket) 형태로 넣기만 해도 충분하다.

--------------------------------------------------

10. 왜 검색용 heap에도 lazy deletion이 필요한가

차량 상태는 계속 바뀐다.

- 주차됨
- 출차됨
- 다시 주차됨
- 견인됨
- 다시 입차됨

그런데 heap은 중간 원소를 바로 삭제하거나 갱신하기 어렵다.

예를 들어 어떤 차량이 parked 상태에서 빠졌다고 해서,
그 차량을 suffix heap 안에서 즉시 제거하기는 어렵다.

그래서 이 문제에서는 heap을 “현재 상태의 원본”으로 사용하지 않는다.

현재 실제 상태는 항상
parked
towed
dict가 가지고 있고,

heap은 빠르게 top 후보를 찾기 위한 보조 구조로만 사용한다.

즉 heap에 예전 기록이 남아 있어도 괜찮다.
대신 꺼낼 때마다
- 지금도 dict에 존재하는지
- ticket이 일치하는지
를 확인하고, 맞지 않으면 버린다.

이 방식이 lazy deletion이다.

--------------------------------------------------

11. _process_tow(mTime)를 모든 API 시작 전에 호출하는 이유

문제의 중요한 조건은 mTime이 항상 증가한다는 것이다.

즉 현재 시각이 mTime이라면,
그 이전에 견인되어야 할 차량은 모두 이미 견인된 상태여야 한다.

그래서 enter, pullout, search 중 어떤 API가 호출되든,
제일 먼저 해야 하는 일은

- tow_heap에서 tow_time <= mTime인 이벤트를 모두 꺼내고
- 실제로 현재 유효한 parked 상태인지 확인한 뒤
- 견인 처리하는 것

이다.

이 과정을 하지 않으면 현재 시각 기준 상태가 틀어지게 된다.

예를 들어 견인되어야 할 차량이 아직 parked에 남아 있다면,
pullout이나 search 결과가 잘못될 수 있다.

즉 _process_tow(mTime)는
현재 시각까지의 상태를 먼저 정상화하는 공통 전처리다.

--------------------------------------------------

12. enter() 알고리즘이 왜 그런 순서로 구성되는가

입차 처리의 실제 사고 흐름은 다음과 같다.

1) 현재 시각까지 견인 처리
먼저 현재 상태를 최신으로 맞춘다.

2) 이 차량이 견인 기록 상태인지 확인
문제 조건상 견인된 차량이 다시 입차하면,
주차 성공 여부와 관계없이 견인 기록은 삭제된다.

따라서 주차 성공 여부를 판단하기 전에 towed 기록부터 제거해야 한다.

3) 주차 위치 선택
- 가장 빈 슬롯이 많은 구역 선택
- 그 구역에서 가장 작은 슬롯 선택

4) parked 등록
- 차량의 현재 주차 상태 저장
- 미래 견인 이벤트 등록
- suffix 검색용 parked heap 등록

즉 enter()는
상태 정리 -> 기존 견인 기록 정리 -> 주차 위치 선정 -> 새 상태 등록
이라는 순서로 구성된다.

--------------------------------------------------

13. pullout() 알고리즘이 왜 그런 구조인가

출차 처리에서 필요한 것은 차량 상태에 따른 분기뿐이다.

1) 현재 시각까지 견인 처리
현재 상태를 최신으로 맞춘다.

2) parked에 있으면 일반 출차
- 슬롯을 비운다
- 주차 기간 mTime - enter_time 반환

3) parked에는 없고 towed에 있으면 견인 차량 처리
- 견인 기록 삭제
- 벌점 포함 반환값 계산

4) 둘 다 아니면 -1
현재 주차 중도 아니고 견인 기록도 없는 차량이므로 실패

즉 pullout()은
현재 주차 중인지 / 견인됐는지 / 아무 상태도 아닌지
를 구분하는 함수다.

--------------------------------------------------

14. search() 알고리즘이 왜 그런 구조인가

검색 조건을 보면 parked가 towed보다 항상 우선이다.

즉 parked와 towed를 한 구조에서 섞어서 처리하기보다,
아예 두 단계로 나누는 것이 더 단순하고 명확하다.

1) 현재 시각까지 견인 처리
검색 시점 상태를 최신으로 맞춘다.

2) suffix에 해당하는 parked 차량에서 최대 5대 수집
이들이 항상 우선순위가 더 높다.

3) 부족하면 같은 suffix의 towed 차량으로 채우기
parked가 5대 미만일 때만 보충

이 구조 덕분에 상태 우선순위를 복잡한 비교로 섞지 않고도
문제 요구를 정확히 구현할 수 있다.

--------------------------------------------------

15. 이 설계가 제약을 만족하는 이유

구역 선택
구역 수 N <= 26이므로 선형 탐색으로 충분하다.

슬롯 선택
각 구역의 슬롯 수 M <= 1000이므로 heap으로 빠르게 최소 슬롯을 선택할 수 있다.

견인 이벤트
입차마다 견인 이벤트 하나가 생기지만,
각 이벤트는 결국 한 번만 실제로 처리되거나 stale 상태로 버려진다.
따라서 전체적으로 amortized 효율이 좋다.

검색
suffix별 heap에서 top 몇 개만 확인하면 되므로,
전체 차량을 모두 훑는 방식보다 훨씬 빠르다.

또한 stale entry는 lazy deletion으로 버려지지만,
각 기록은 전체적으로 한두 번만 무효 처리되므로 감당 가능하다.

--------------------------------------------------

16. 이 풀이의 핵심 철학

이 문제에서 가장 중요한 설계 철학은 다음 세 가지다.

1) 현재 실제 상태와 보조 인덱스를 분리한다
- 실제 상태의 원본: parked, towed
- 빠른 검색과 이벤트 처리를 위한 보조 구조: 각종 heap

즉 “지금 진짜 상태는 dict가 들고 있고,
heap은 빠른 접근을 위한 보조 도구일 뿐”이라고 생각해야 한다.

2) 작은 축은 단순하게, 우선순위가 필요한 축만 heap으로 처리한다
- 구역 수는 작으므로 배열 + 선형 탐색
- 슬롯 번호는 최소값 추출이 필요하므로 heap
- 견인 시각은 미래 이벤트 우선순위가 필요하므로 heap
- suffix 검색은 top 차량 추출이 필요하므로 heap

3) 시간 증가 조건을 적극적으로 이용한다
mTime이 항상 증가하기 때문에 과거 상태로 되돌아갈 필요가 없고,
현재 시각까지의 이벤트만 순서대로 반영하면 된다.

이 조건 덕분에 복잡한 상태 롤백 없이
단순한 이벤트 시뮬레이션으로 문제를 해결할 수 있다.

17. 최종 요약

왜 이런 자료구조를 선택했는가

- 차량 번호로 현재 상태를 즉시 찾기 위해 parked, towed dict 사용
- 구역 선택은 N이 작아서 free_cnt 선형 탐색 사용
- 구역 내 최소 슬롯 선택을 위해 free_slots heap 사용
- 자동 견인은 미래 이벤트이므로 tow_heap 사용
- suffix 검색을 빠르게 하기 위해 park_suffix, tow_suffix 사용
- 재입차/재견인으로 과거 기록이 남을 수 있으므로 ticket 기반 lazy deletion 사용

왜 이런 알고리즘으로 구성했는가

- 모든 API 시작 전에 현재 시각까지의 견인을 먼저 처리
- enter()는 상태 정리 후 견인 기록 제거, 슬롯 선정, parked 등록
- pullout()은 parked / towed / 없음 세 상태로 분기
- search()는 parked 먼저, 부족하면 towed로 채우는 방식으로 구현
- heap 안의 오래된 기록은 dict와 ticket으로 검증해서 무시

이렇게 설계하면 문제의 모든 조건을 자연스럽게 만족하면서도,
제약 안에서 충분히 빠르게 동작하는 풀이가 된다.
